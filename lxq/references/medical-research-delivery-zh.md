## 一、模块定位

本文件是 LXQ 中“医学科研项目客户交付”轨道的独立中文规则。用于生成医院、科室、科研客户可以直接阅读的课题筛选、售前方案或基金申报初稿，同时保留研究设计、证据来源和待确认事项的审计记录。

本文件不是英文版的零散翻译，而是中文任务的默认执行规范。中文请求优先读取本文件；只有用户明确要求英文交付时，才读取 `medical-research-delivery.md`。

## 二、内部文件对应关系

| 文件 | 内部作用 |
|---|---|
| `medical-research-delivery-zh.md` | 中文客户交付规则，即本文件 |
| `medical-research-delivery.md` | 英文客户交付规则 |
| `grant-qc.md` | 基金指南、科学问题、研究目标、风险与预算审查 |
| `output-schemas.md` | 结构化表格和字段定义 |
| `scaffold_review.py` | 生成基金工作包、研究设计表和客户方案模板 |
| `validate_grant_delivery.py` | 检查方案章节和研究设计字段是否完整 |
| `validate_review_bundle.py` | 检查整个 LXQ 交付包的文件与表头结构 |
| `score_delivery_quality.py` | 对中文客户交付进行结构、语言和预算启发式评分 |
| `score_grant_quality.py` | 对基金方案进行目标、方法、证据和预算启发式评分 |
| `static/fragments/invocation_intensity.md` | 根据任务选择 light、standard、strict 或 forensic |
| `static/fragments/style/anti_ai_zh.md` | 中文正式交付的去AI味语言后处理 |
| `references/output_contracts/customer_delivery.md` | 客户交付任务的专用输出契约 |

中文基金客户方案通常同时启用 `grant` 与 `delivery` 两条轨道：前者保证科学与申报合规，后者保证客户可读和直接交付。

## 三、适用范围

出现以下任一情况时使用本模块：

- 用户提供医院、科室、研究团队或客户背景；
- 用户提供基金类型、申报指南、研究周期、经费上限或评审要求；
- 用户提供疾病方向、临床问题、实验技术、生信技术或拟研究机制；
- 用户要求筛选课题方向、形成售前方案或撰写基金初稿；
- 用户要求内容正式、结构完整、细节明确、客户直接可读；
- 用户明确要求减少模板化表达或“AI 空话”。

## 四、禁止和边界

不得将本模块用于：

- 代替真实临床诊疗建议；
- 代替伦理委员会、统计师、专科专家或基金管理部门的最终审查；
- 虚构指南、文献、前期结果、伦理批件、样本资源、设备条件或合作关系；
- 将拟开展的实验写成已经完成；
- 将相关性研究写成因果研究；
- 在没有直接机制证据时宣称机制已经被证明；
- 凭记忆断言当前申报截止时间、资格、篇幅或经费政策。

## 五、输入信息清单

开始前尽量明确：

1. 客户类型、医院和科室；
2. 客户现有研究基础、病例资源、样本资源、数据资源和实验平台；
3. 基金名称、申报年份、指南原文、评审方向和截止时间；
4. 项目周期、经费上限和可承担的主要成本；
5. 疾病方向、目标人群、核心临床或科学问题；
6. 拟采用的湿实验、生信、影像、组学或人工智能技术；
7. 用户需要课题筛选、客户方案、正式标书还是其中某个章节；
8. 期望交付格式、篇幅、语言和截止时间。

缺失但不会改变研究方向的信息，可使用“建议”“拟采用”或“需进一步确认”。缺失且会改变研究设计的信息，必须标记 `[AUTHOR_INPUT_NEEDED: 具体问题]`；尚未核验的申报政策标记 `[POLICY_CHECK_NEEDED: 具体政策]`。

## 六、执行流程

### 1. 建立客户与基金简报

先形成一页事实底稿，区分以下五类内容：

- 已核验的公开事实；
- 客户明确提供的信息；
- 已检查的客户材料或前期证据；
- 拟开展的研究工作；
- 假设、推断和待确认事项。

### 2. 筛选课题方向

当研究方向尚未确定时，先给出候选方向，再写完整方案。每个方向必须说明：

- 拟定题目；
- 一句话核心科学问题或研究假设；
- 与客户科室、病例资源和技术平台的匹配度；
- 与基金指南和经费规模的匹配度；
- 临床或科学意义；
- 相对于最接近研究的创新点；
- 样本、周期、人员和技术可行性；
- 主要成本、关键风险和替代方案；
- 建议等级：`优先推荐`、`备选` 或 `不建议`。

方向筛选必须使用下方“候选课题方向评分规则”。评分是项目内部决策工具，不得包装成通用学术标准。

### 3. 建立证据边界

使用已核验文献支撑疾病负担、研究现状、知识缺口和方法依据。不得仅凭题名或摘要片段声称文献支持某个具体机制。需要系统检索时调用 `nature-academic-search`，需要补充和核验引用时调用 `nature-citation`，需要全文精读时调用 `nature-reader`。

### 4. 先完成研究设计，再润色文字

每个入选方向必须明确：

- 研究对象、研究场景、研究设计和实验单位；
- 样本量目标、计算或估算依据；
- 研究分组、对照设置、随机化和盲法（适用时）；
- 入组标准和排除标准；
- 主要终点、次要终点和观察时间点；
- 样本或数据类型、检测指标和质量控制；
- 统计方法、缺失数据、多重比较和敏感性分析；
- 验证策略、成功标准和失败后的替代路线；
- 预期结果及其证据边界。

无法完成正式样本量计算时，应写清计算框架、需要的效应量或事件率、显著性水平、把握度或精度目标、失访率和最终待确认参数。不得只给一个方便执行但没有依据的数字。

### 5. 完成湿实验方案

湿实验至少写明：

- 样本来源、采集、保存、运输和前处理；
- 实验平台、试剂或检测方法；
- 阳性对照、阴性对照、空白对照及适用的基准样本；
- 生物学重复和技术重复；
- 随机化、盲法和批次平衡；
- 质量控制指标、通过阈值和失败判定；
- 主要检测指标和次要检测指标；
- 正交验证或独立方法验证；
- 实验失败时的替代方法。

### 6. 完成生信方案

生信方案至少写明：

- 自有数据、公共数据或联合数据的来源和访问状态；
- 数据质控、预处理、标准化和批次效应处理；
- 参考基因组、注释版本、特征定义和标识符体系；
- 核心算法、软件版本、参数和随机种子；
- 训练集、验证集和测试集的划分；
- 数据泄漏、过拟合和类别不平衡的控制；
- 内部验证、外部队列验证、临床验证或湿实验验证；
- 效应量、置信区间、模型性能和校准指标；
- 可解释性、稳健性和敏感性分析；
- 计划输出的流程图、质控图、差异图、模型图、验证图和机制图。

不得只罗列热门算法。每个算法都要说明输入数据、服务的研究目标、主要输出及验证方法。

## 七、候选课题方向评分规则

当用户要求“给我几个方向”“选三个最好的”“兼顾创新性和可行性”时，必须对候选方向进行内部评分。

评分维度：

1. 客户匹配度：0–5分
2. 临床问题明确度：0–5分
3. 创新性：0–5分
4. 可行性：0–5分
5. 经费匹配度：0–5分
6. 数据/样本可获得性：0–5分
7. 发表或申报潜力：0–5分
8. 风险可控性：0–5分

总分解释：

- ≥32分：优先推荐
- 26–31分：可作为备选
- <26分：不推荐，除非用户明确坚持

输出要求：

- 面向用户时可以不展示完整评分表，但必须基于评分选择推荐方向。
- 如果用户要求“为什么选这三个”，需说明筛选理由。
- 不得只按创新性排序，必须同时考虑可行性、经费、样本和客户背景。
- 缺少客户、样本或预算信息时，相应维度不得默认给满分，应标记假设并降低确定性。

## 八、经费—技术复杂度匹配规则

### ≤3万元

优先采用：

- 回顾性临床资料分析
- 小样本前瞻性观察
- 常规检验指标
- 临床量表
- 影像定量
- qPCR
- IHC
- ELISA
- 组织芯片小规模验证
- 简单机器学习模型

避免默认设计：

- 单细胞测序
- 空间转录组
- 大规模代谢组
- 大规模蛋白组
- 多中心前瞻性大队列
- 完整动物机制验证
- 类器官大规模筛选

### 3–5万元

可加入：

- 小规模机制验证
- 细胞实验
- 靶向检测
- 组织芯片
- 基础风险模型
- 小规模外部验证

### 5–10万元

可加入：

- 多时间点检测
- 组学子队列
- 动物或类器官探索
- 较完整机制验证
- 多模态数据整合

### >10万元或重点项目

可设计：

- 多中心队列
- 多组学整合
- AI模型
- 空间组学
- 单细胞组学
- 系统机制验证
- 转化应用路径

规则：

- 经费≤3万元时，除非用户明确要求，不默认写高成本组学和动物实验。
- 如果用户要求高创新但经费很低，优先设计“临床表型 + 常规指标 + 小规模验证 + 风险模型”的路径。
- 所有技术路线必须与经费、周期、平台和样本获得难度匹配。
- 上述区间是初步设计约束，不替代地区报价、机构收费和正式预算核算。

## 九、中文客户交付默认结构

当用户要求“客户直接看”“完整售前”“交付版本”“正式一点”“方案内容”“申报书内容”时，默认输出以下结构：

1. 拟定题目
2. 研究背景与临床问题
3. 研究假说
4. 研究目标
5. 主要研究内容
6. 拟解决的关键问题
7. 研究对象与样本量
8. 分组、检测指标与随访节点
9. 统计学与模型构建
10. 技术路线
11. 创新点
12. 可行性分析
13. 风险与替代方案
14. 预期成果

硬性要求：

- 每个研究内容必须包含研究对象、方法、指标和预期判断。
- 临床研究必须明确研究类型、样本量、入排标准、分组方式、主要终点、次要终点、检测指标和统计方法。
- 生信研究必须明确数据来源、预处理、算法、参数、训练/验证策略、防泄漏措施、评价指标和可视化结果。
- 湿实验研究必须明确样本来源、处理流程、对照、重复、批次、QC、正交验证和替代方案。
- 不明确的信息必须使用“拟采用”“建议”“计划”“需进一步确认”，不得写成既定事实。
- 不得虚构前期基础、伦理编号、样本量、检测结果、参考文献或合作平台。

若官方模板与本结构冲突，以当前官方模板为准，但必须在内部研究设计表中保留样本量、入排标准、终点、统计、验证、风险和预算映射。

## 十、各部分写作要求

### 一、拟定题目

题目应包含研究对象、关键变量或干预、核心结局或机制层级；避免范围过大、技术堆砌和无法验证的因果表述。

### 二、研究背景与临床问题

按照“疾病或临床问题—现有证据—关键不足—拟解决问题—研究价值”展开。每个重要事实应有可核验来源。

### 三、研究假说

写成可证伪的关系或机制假说，明确研究对象、关键变量、方向和验证方式；证据不足时不得把假说写成事实。

### 四、研究目标

设置一个总体目标和若干具体目标。目标必须可测量，并与研究内容、终点、统计方法和预期成果一一对应。

### 五、主要研究内容

以工作包或研究任务组织；每项必须说明研究对象、方法、指标、预期判断、输入、输出、里程碑、依赖关系和替代路线。

### 六、拟解决的关键问题

聚焦真正限制领域进展或临床应用的问题，不得把一般技术步骤包装成关键科学问题。

### 七、研究对象与样本量

写明研究类型、样本量及依据、入排标准、招募或样本获取方式，以及失访、失败样本和分析集的处理。

### 八、分组、检测指标与随访节点

写明分组方式、对照、随机化/盲法（适用时）、主要和次要检测指标、主要和次要终点、基线及随访时间点。

### 九、统计学与模型构建

写明描述性统计、主要分析、协变量、效应量和置信区间、缺失数据、多重比较、敏感性分析、内部和外部验证。预测模型需报告区分度、校准度和临床效用。

### 十、技术路线

描述从研究对象或数据获取、实验和计算分析、验证、解释到成果输出的完整路径，并标明关键质控点和失败后的分支。必要时给出流程图。

### 十一、创新点

围绕新问题、新人群、新数据、新方法、新验证链或新转化价值表述。避免未经检索支持的“首次”“首创”或“填补空白”。

### 十二、可行性分析

从病例或样本、数据、平台、人员、周期、经费、伦理、协作和风险预案进行论证。不得虚构客户条件。

### 十三、风险与替代方案

逐项说明招募、样本质量、实验失败、批次、数据缺失、模型过拟合、外部验证和经费超支风险，以及预警、缓解、替代路径和残余风险。

### 十四、预期成果

区分科学结果、过程里程碑、交付物和长期影响。不得保证获得阳性结果、证明机制或发表特定等级论文。

### 参考文献（需要时）

仅列入已核验题名、作者、期刊、年份、卷期页码和 DOI 或 PMID 的文献。保持正文观点与引用逐条对应，不得虚构文献。

## 十一、语言与交付原则

1. 优先输出可以直接给客户阅读的版本，不以泛泛建议代替交付物。
2. 每个方向必须同时考虑创新性、可行性、客户背景和经费匹配。
3. 表达正式、具体、克制，删除口号、套话、重复过渡和无信息量总结。
4. 不确定内容使用“建议”“拟采用”“计划”“可能”“需进一步确认”。
5. 明确区分相关、预测、因果、机制和临床效用。
6. 预期结果写成可测量结果和决策标准，不写成必然实现的承诺。
7. 题目、目标、研究内容、方法、终点、统计、图表、里程碑和预算应保持一一对应。
8. 中文正式交付前默认应用 `static/fragments/style/anti_ai_zh.md`，但不得因语言压缩而删除证据边界、限制或待确认项。

## 十二、交付前质量门槛

逐项检查：

- [ ] 是否明确客户背景、基金类型、研究周期和经费边界？
- [ ] 每项研究内容是否包含研究对象、方法、指标和预期判断？
- [ ] 是否给出明确样本量及其依据？
- [ ] 是否列明入组标准和排除标准？
- [ ] 是否明确主要终点和次要终点？
- [ ] 是否明确分组、对照、时间点和检测指标？
- [ ] 湿实验是否包含对照、重复、批次质控和正交验证？
- [ ] 生信是否包含数据来源、核心算法、验证策略、性能指标和可视化？
- [ ] 统计方法是否与研究设计、实验单位和终点匹配？
- [ ] 是否处理缺失数据、多重比较、数据泄漏和过拟合？
- [ ] 是否避免因果和机制的过度推断？
- [ ] 是否与样本通量、平台、人员、周期和经费规模匹配？
- [ ] 是否显式标记未经核验或需要客户确认的信息？
- [ ] 是否删除“AI 空话”和模板化套话？
- [ ] 是否能让客户不依赖额外解释直接阅读？
- [ ] 参考文献是否真实、可追溯并与正文观点对应？

任何必需项缺失时，不得标记为 `ready_to_use`。根据情况使用 `conditional`、`needs_author_input` 或 `blocked`。

## 十三、验证与交付

生成基金工作包：

```powershell
python scripts/scaffold_review.py --profile grant --output <输出目录>
```

检查工作包结构：

```powershell
python scripts/validate_review_bundle.py <输出目录> --profile grant
```

检查客户方案和研究设计是否达到交付要求：

